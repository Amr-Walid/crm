## 1. الهيكل العام للمشروع (Clean Architecture)

بدلاً من وضع كل الكود في مشروع واحد (مما يجعله معقداً وصعب الصيانة بعد فترة)، نقوم بتقسيم المشروع إلى **4 مشاريع فرعية (Layers)** داخل Solution واحد. 

القاعدة الذهبية هنا هي: **"المشاريع الداخلية لا تعرف شيئاً عن المشاريع الخارجية"**. الدائرة الداخلية (Domain) هي قلب النظام ولا تعتمد على أي مكتبات خارجية.

```
                  ┌──────────────────────────────┐
                  │          API (Web)           │
                  └──────────────┬───────────────┘
                                 │ (تعتمد على)
                  ┌──────────────▼───────────────┐
                  │        Infrastructure        │
                  └──────────────┬───────────────┘
                                 │ (تعتمد على)
                  ┌──────────────▼───────────────┐
                  │         Application          │
                  └──────────────┬───────────────┘
                                 │ (تعتمد على)
                  ┌──────────────▼───────────────┐
                  │            Domain            │
                  └──────────────────────────────┘
```

### أ. الطبقة الأولى: Domain (قلب النظام)
* **ماذا تحتوي؟** الكيانات الأساسية (Entities) التي تمثل جداول قاعدة البيانات مثل (`Customer`, `Device`, `Ticket`) والـ Enums.
* **الاعتمادية:** لا تعتمد على أي مشروع آخر ولا تحتوي على أي مكتبات خارجية (خالية تماماً من الـ EF Core أو الـ SQL Server).
* **مثال:** كلاس يمثل العميل:
  ```csharp
  public class Customer
  {
      public int Id { get; set; }
      public string Name { get; set; } = string.Empty;
      public string Phone { get; set; } = string.Empty;
  }
  ```

### ب. الطبقة الثانية: Application (منطق العمل - Business Logic)
* **ماذا تحتوي؟** العمليات التي تتم داخل النظام (مثل: إنشاء تذكرة جديدة، تحديث حالة ضمان)، والـ Interfaces، والـ Validation.
* **الاعتمادية:** تعتمد **فقط** على الـ Domain.
* **الوصف:** هنا نكتب "ماذا يفعل السيستم" دون الاهتمام بكيفية تخزين البيانات أو من أين يأتي الطلب.

### ج. الطبقة الثالثة: Infrastructure (البنية التحتية والربط الخارجي)
* **ماذا تحتوي؟** الاتصال الفعلي بقاعدة البيانات (DbContext)، التعامل مع الملفات (S3/Local)، إرسال الإيميلات والرسائل.
* **الاعتمادية:** تعتمد على الـ Application والـ Domain.
* **الوصف:** هنا نربط النظام بالعالم الخارجي. لو أردنا تغيير قاعدة البيانات من SQL Server إلى PostgreSQL مثلاً، سنقوم بالتغيير في هذه الطبقة فقط دون لمس باقي الأقسام.

### د. الطبقة الرابعة: API / Presentation (الواجهة المفتوحة للعالم)
* **ماذا تحتوي؟** الـ Controllers، والـ Middlewares، وإعدادات الـ Authentication وتوزيع الـ Routes.
* **الاعتمادية:** تعتمد على كل الطبقات السابقة لتقوم بجمعهم وتشغيل البرنامج.
* **الوصف:** تستقبل الطلب (Request) من المتصفح أو تطبيق الهاتف، وتمرره للـ Application لمعالجته، ثم تعيد النتيجة (Response) كـ JSON.

## 2. نمط فصل القراءة عن الكتابة (CQRS)
* **CQRS** تعني **Command Query Responsibility Segregation**.
* **الفكرة:** فصل العمليات التي **تُغير** البيانات (إضافة، تعديل، حذف) عن العمليات التي **تقرأ** البيانات فقط (استعلام، فلترة).

### لماذا نستخدمها؟
في الأنظمة الكبيرة، القراءة تكون متكررة جداً وتحتاج لسرعة عالية (مثل لوحة التحكم)، بينما الكتابة تحتاج لعمليات تحقق وتدقيق صارمة (Validation). فصلهم يسهل صيانة الكود ويحسن الأداء.

* **الـ Command (الأمر):** يمثل عملية تغيير (مثال: `CreateTicketCommand`).
* **الـ Query (الاستعلام):** يمثل عملية قراءة (مثال: `GetTicketByIdQuery`).

## 3. الوسيط (MediatR) وكيف يربط الكود

مكتبة **MediatR** هي تطبيق لنمط تصميم اسمه **Mediator Pattern**.
* **المشكلة:** بدونها، الـ API Controller سيحتاج لعمل Inject لـ 10 خدمات (Services) مختلفة لكي يؤدي وظيفته، مما يجعل الكود متشابكاً وصعب التعديل.
* **الحل:** الـ Controller يرسل رسالة واحدة فقط للـ MediatR (الوسيط)، والـ MediatR يبحث عن الكود المسؤول عن تنفيذ هذه الرسالة ويشغله.

### مثال عملي لدورة الطلب مع MediatR و CQRS:

1. يأتي طلب من الموبايل لإنشاء عميل جديد إلى الـ Controller:
   ```csharp
   [HttpPost]
   public async Task<IActionResult> Create(CreateCustomerCommand command)
   {
       // نرسل الأمر للوسيط فقط دون أن نعرف من سينفذه
       var result = await _mediator.Send(command); 
       return Ok(result);
   }
   ```
2. يستقبل MediatR الـ `CreateCustomerCommand` ويوجهها تلقائياً للـ Handler الخاص بها في طبقة الـ **Application**:
   ```csharp
   public class CreateCustomerCommandHandler : IRequestHandler<CreateCustomerCommand, int>
   {
       private readonly ICustomerRepository _repository;
       // Constructor لعمل Inject للمخزن فقط
       
       public async Task<int> Handle(CreateCustomerCommand request, CancellationToken cancellationToken)
       {
           // هنا نكتب التحقق وحفظ البيانات
           var customer = new Customer { Name = request.Name, Phone = request.Phone };
           await _repository.AddAsync(customer);
           return customer.Id;
       }
   }
   ```

## 4. التعامل مع قاعدة البيانات (EF Core Code-First)

سنستخدم أسلوب **Code-First** (الكود أولاً):
1. نقوم بكتابة كلاسات عادية (Entities) في مشروع الـ Domain تمثل الجداول التي نريدها.
2. نقوم بتهيئة الـ `DbContext` في مشروع الـ Infrastructure وتحديد العلاقات بين الجداول.
3. نستخدم الـ **EF Core CLI** لعمل **Migration** (ترحيل): يقوم النظام بقراءة الكود وتحويله تلقائياً لـ SQL Scripts وتطبيقها على قاعدة البيانات في الـ SQL Server.

### الفائدة:
أي تعديل في الجداول (إضافة عمود، تغيير علاقة) يتم داخل الكود أولاً، والـ Migration يتكفل بنقل التعديل لقاعدة البيانات دون كتابة أوامر SQL يدوية معقدة.

## 5. الحماية والصلاحيات (Identity & Permission-Based Auth)

النظام سيحتوي على مستويات أمان متقدمة تناسب الشركات الكبرى:

### أ. تسجيل الدخول بالـ JWT والـ Refresh Tokens
* **JWT (JSON Web Token):** بعد تسجيل دخول الموظف، يعطيه السيستم رمزاً (Token) مشفراً يحتوي على بياناته. هذا الرمز يرسله الموظف مع كل طلب. لزيادة الأمان، مدة صلاحية هذا الرمز تكون قصيرة جداً (مثلاً 15 دقيقة).
* **Refresh Token:** عبارة عن رمز آخر طويل الصلاحية (مثلاً 7 أيام) يتم تخزينه في قاعدة البيانات بأمان. قبل انتهاء الـ JWT، يرسل المتصفح الـ Refresh Token للحصول على JWT جديد دون أن يضطر الموظف لإعادة كتابة اسم المستخدم وكلمة المرور.

### ب. الصلاحيات المبنية على التصاريح (Permission-Based Authorization)
في الأنظمة العادية، يتم التحقق من الصلاحيات بناءً على الدور (Role-Based) مثل: `[Authorize(Roles = "Agent")]`.
* **المشكلة:** لو قررت الإدارة فجأة السماح لبعض موظفي خدمة العملاء (Agents) بإغلاق الحالات، ومنع البعض الآخر، ستضطر لتعديل الكود أو إنشاء أدوار جديدة معقدة.
* **الحل الاحترافي:** نتحقق بناءً على تصريح محدد (Permission) مثل: `[Authorize(Policy = "CanCloseTicket")]`.
* **كيف تعمل؟:** 
  1. ننشئ جدولاً للتصاريح (مثلاً: `CanCreateTicket`, `CanDeleteCustomer`, `CanCloseTicket`).
  2. نربط الموظف بالتصاريح الفردية المسموحة له.
  3. نقوم بالتحقق من هذا التصريح في الكود. هكذا يستطيع الـ Admin تغيير صلاحيات أي موظف من لوحة التحكم فوراً دون تعديل سطر كود واحد!

## 6. خارطة الطريق العملية لتنفيذ المشروع

لتجنب تشتت الكود وظهور أخطاء برمجية، سنقوم بالعمل خطوة بخطوة عبر **المراحل الستة** المحددة في الخطة. في كل مرحلة سنقوم بالآتي:
1. **تصميمEntities وقاعدة بيانات الـ Module** الخاصة بالمرحلة فقط.
2. **برمجة الـ Commands والـ Queries** لتغطية العمليات المطلوبة.
3. **إنشاء الـ API Controllers** واختبارها عبر Swagger للتأكد من أنها ترجع بيانات JSON صحيحة.
4. **تأكيد جاهزية الـ Module** والانتقال للمرحلة التالية.

هذا يضمن أننا سنبني نظاماً احترافياً خالياً من التعقيدات وبأعلى جودة برمجية ممكنة!
